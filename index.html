01 — Work Project

WALALAND 상품등록 성능 개선기

  • 기간 : 2024.09 ~ 2024.12
  • 역할 : Backend Developer
  • 성과 : 약 11배 성능 개선 달성

최종 결과

API 전체 소요 시간 ~13분 → ~1분
파파고 API 호출 수 3,922회 → ~92회
DB 쿼리 횟수 5,230+ → ~53회

기존 문제점

  • 1+1 이벤트 상품의 경우 옵션1(37개) × 옵션2(53개) = 1,961개의 옵션 데이터를 저장 필요
  • 옵션마다 파파고 번역 API를 2회씩 호출 → 약 3,922회의 외부 API 호출 발생
  • DB에도 4/3 × N번의 Insert + Update 쿼리가 개별 실행 → 약 5,230번 이상 쿼리 실행
  • 전체 소요 시간 총 약 13분

개선 과정

1
원인 파악 ① — 중복 번역 API 호출
옵션명/옵션값 데이터를 해외 유저 대응을 위해 파파고 API로 번역하고 있었으나, 중복된 값(색상·사이즈 등)도 매번 개별 호출하고 있었습니다.
2
개선 ① — HashMap 기반 로컬 캐싱 적용
유니크한 옵션명/옵션값만 번역하도록 HashMap 로컬 캐싱을 적용. 번역 대상이 1,961개 → 약 92개로 감소. 파파고 API 호출 시간 약 770초 → 약 40초, 전체 소요 시간 약 13분 → 약 1분 50초로 단축.
3
원인 파악 ② — 대량 개별 DB 쿼리
N개 옵션 등록 시 약 4/3 × N번의 Insert + Update가 개별 실행. 문제 상품의 경우 약 5,230번 이상의 쿼리가 실행되고 있었습니다.
4
개선 방안 비교 — 비동기 vs Bulk 처리
Auto_Increment의 테이블 수준 Lock으로 비동기 Insert 시 병목이 예상되어 Bulk 처리를 선택. DB 접근 수 자체를 줄여 리소스도 절감.
방안선택사유
도메인/DB/API 설계 개편보류프론트 대대적 수정 필요, 당시 프론트 파트 바쁜 상황
IO 비동기 처리미채택Auto_Inc Lock으로 병목 예상 + 추가 스레드 리소스
Bulk 처리 도입✓ 채택DB 접근 수 자체를 줄여 통신 횟수 + 리소스 모두 절감
5
트러블슈팅 — Hibernate Identity 채번 시 Bulk Insert 미지원
Identity(Auto_Increment) 방식에서는 Hibernate의 쓰기지연이 동작하지 않아 Bulk Insert 불가. Table 채번 방식은 채번 부하가 더 커 성능 저하. JDBC를 직접 사용하여 해결.
6
트러블슈팅 — rewriteBatchedStatements 옵션
JDBC batchUpdate 후에도 개별 Insert가 발생. MySQL Connector/J의 rewriteBatchedStatements 옵션(기본 false)을 true로 설정하여 Multi Value Insert로 변환.
7
트러블슈팅 — innodb_autoinc_lock_mode & binlog_format
interleaved lock mode(mode=2) 설정으로 테이블 수준 Lock 제거. binlog_format을 MIXED → ROW로 변경하여 replication 보안성을 확보하고자 하였습니다. (MySQL 8.0 이후 기본값)

회고

중복된 API 호출을 줄이고, Batch 처리를 도입하여 약 11배의 성능 개선에 성공했지만 크게 두 가지 아쉬운 점이 있습니다.

1. 번역할 데이터량이 훨씬 많아질 경우

  • 옵션 데이터는 서로 다른 상품일 경우에도 사이즈, 색상과 같이 지속적으로 중복해서 사용될 가능성이 높다는 특성을 갖고 있습니다. 따라서 현재 번역 결과물을 메모리 기반 캐싱 구조를 적용했지만, DB에서 영속적으로 보관해둘 가치가 충분히 존재한다고 느꼈습니다. 아예 사전 시스템과 연계 설계하여 재사용할 수 있는 구조로 확장도 추후에 고려해볼 수 있을 것 같습니다.
  • 순수 번역해야할 데이터량이 많아질 경우 파파고 API를 호출하는 로직을 비동기로 처리해 처리 시간을 단축시킬 수 있을 것 같습니다.

2. 외부 시스템에 대한 의존성

  • 현재 구조로는 상품등록이 완료되기까지 파파고 시스템에 강하게 의존하고 있습니다. 파파고 시스템에 장애가 발생하거나, 병목이 발생할 경우 그것이 서버에 그대로 전파될 수 있다는 문제가 존재합니다.
  • 상품이란 데이터는 사용자에게 노출되기 전 준비가 완료되면 된다는 특성이 있습니다. 따라서 "번역완료" 등의 상태를 추가하고 번역완료된 상품에 대해서만 외부에서 조회될 수 있도록 한다면, 상품등록 기능을 비동기 아키텍처 구조로 설계가 가능할 것 같습니다. 즉, 번역로직을 상품등록 이후 시점에 수행되도록하여 관리자에게 파파고 시스템과 상관없이 즉시 응답하는 구조입니다.
  • 이렇게되면 관리자는 상품등록이 번역과 함께 온전히 완료될 때까지 기다릴 필요없이 즉시 응답을 받고, 다른 작업에 집중할 수 있어 업무 효율성도 증대할 수 있을 것입니다.
  • 하지만 번역 데이터량이 소량이거나 재사용만으로 가능할 경우, 이 구조는 오히려 비효율을 초래할 수도 있을 것입니다. 이는 등록할 상품에 따라 관리자가 판단이 가능하므로 일반 상품 등록과, 비동기 상품 등록을 구분하여 제공하는 방법도 있을 것 같습니다. 혹은 추후에 상품 대량 등록 기능 구현 시, 이 방식을 고려할 수 있을 것 같습니다.

백엔드 핵심 구현

01 클린 아키텍처 리펙토링
DB 교체 경험을 계기로 도메인의 외부 독립성 확보를 위해 도입
Domain → UseCase → Port → Adapter 4개 계층으로 구성
DIP를 활용해 Adapter와 UseCase 간 느슨한 결합 확보
Domain/Port 계층은 외부 라이브러리 의존성 없음
02 도메인 주도 개발 (DDD)
순수 도메인 객체(Value Object)와 영속성 테이블 객체(Raw Field) 분리
애그리거트 간 간접 참조 — 독립성 확보 및 외래키 오버헤드 방지
도메인 간 연계 처리 시 Event 활용으로 결합도 최소화
도메인 객체 생성자 Private + withId/withoutId 팩토리 메서드로 완전 객체 보장
03 테스트 주도 개발 (TDD)
Repository / Service / Controller 레이어별 단위 테스트
Mocking 활용 관심사 분리 + @ParameterizedTest 동적 Validation
테스트 헬퍼 모듈화로 가독성 및 유지보수성 향상
04 N+1 문제 설계 수준 해결
간접 참조 DDD 설계로 N+1 문제 자체가 발생하지 않는 구조
연관 엔티티 필요 시 From절 자식 + fetch join, 조회 전용은 DTO 프로젝션
과거 프로젝트에서 BatchSize 적용으로 N+1 → 1+1 최적화 경험

2Gradle로 Jar 빌드 → Docker 이미지 빌드
3Docker Hub에 이미지 Push
4Production 서버에서 Pull & Run
5Nginx 리버스 프록시로 Blue/Green 전환 (무중단)
