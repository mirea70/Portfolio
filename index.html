6
트러블슈팅 — Hibernate Identity 채번 시 Bulk Insert 미지원
"Hibernate disables insert batching at the JDBC level transparently if you use an identity identifier generator."
— Hibernate 공식 문서
JPA의 구현체인 Hibernate에서는 채번 방식이 Identity(DB_Auto Increment)일 경우 Bulk Insert를 지원하지 않습니다. Hibernate는 쓰기지연 방식으로 동작하여, 채번할 ID 값들을 미리 얻어온 후 엔티티들에 할당해야 하는데, Identity 방식으로는 쿼리 실행 전 미리 ID를 조회해 할당이 불가능하기 때문입니다.
6-1
Bulk 처리 방식 결정 — Table 채번 변경 vs JDBC 사용
Bulk 처리 도입을 위해, 크게 두 가지 방법을 고려할 수 있었습니다.
방안 1. 채번 방식을 변경한다
1-1. Table 채번으로 변경 미채택
+ Hibernate Batch Insert 활용 가능
− 기존 auto_increment 기반 데이터의 마이그레이션 필요. 팀에 합류한지 얼마 안 된 상황에서 기존 아키텍처를 변경하는 리스크가 큼
1-2. API 서버에서 대량 채번 후 직접 할당 미채택
+ SnowFlake 등의 알고리즘으로 API 로직 상에서 채번 후 ID를 직접 할당 가능
− Table 채번과 마찬가지로 기존 아키텍처 변경에 따른 리스크. 채번 로직의 추가 복잡성 도입
방안 2. Hibernate를 거치지 않고 DB에 직접 접근하는 기술을 사용한다
2-1. Jooq 도입 미채택
+ Java 코드로 타입 안정성 있게 SQL 작성 가능
− JPA와 같이 어느정도의 러닝커브 예상. 하나의 Bulk 기능만을 위해 신규 기술 도입이 과연 효율적인지 의문
2-2. JDBC 사용 ✓ 채택
+ JDK 기본 포함, 낮은 러닝커브, DB 드라이버 저수준 직접 제어 가능
− 문자열 기반 쿼리로 타입 안정성 없음 — 단일 기능 적용 시 큰 단점 아님
러닝커브와 리스크가 가장 적으면서, 퍼포먼스 기대가 가능한 JDBC 사용 채택
7
트러블슈팅 — MySQL Auto_Increment 핸들링
테이블 단위 Lock으로 인한 병목 문제가 실제로 괜찮을지 공식문서 및 레퍼런스 조사를 진행했습니다. MySQL 공식 문서를 확인해보니 Auto_Increment 핸들링에 대한 내용이 자세하게 나와있었습니다.
innodb_autoinc_lock_mode는 총 0, 1, 2의 3개가 존재합니다.
mode=0 (traditional) — 전통 Lock Mode로서 Table 수준의 Lock을 걸어서 한 Insert문 수행이 끝날 때까지 다른 Insert 작업은 기다려야합니다.
mode=1 (consecutive) — Bulk Insert 시에는 여전히 Table 수준의 Lock을 건다고 합니다. 하지만 삽입할 행 수를 미리 알 수 있는 Simple Inserts의 경우 ID 할당 프로세스 동안만 잠금을 거는 경량 잠금인 mutex를 활용하여 Table 수준의 Lock을 거는 것은 피할 수 있다고 합니다.
mode=2 (interleaved) — Table 수준의 Lock을 사용하지 않습니다. 또한 고유값도 가진다고 하니, Key값에 대한 정합성 문제도 발생하지 않습니다. 하지만 여러 Insert문이 동시에 실행되기 때문에, 생성되는 데이터들의 Key 값들은 3, 7, 8, 10 과 같이 연속적이지 않을 수 있다고 합니다.
MySQL 8.0 버전 이후로는 innodb_autoinc_lock_mode=2가 Default로 변경되어, 테이블 수준 Lock으로 인한 병목은 기본적으로 발생하지 않음을 확인하였습니다.
8
트러블슈팅 — rewriteBatchedStatements 옵션
JDBC batchUpdate를 적용했으나 여전히 개별 Insert가 발생했습니다. MySQL Connector/J의 rewriteBatchedStatements 옵션(기본 false)을 true로 설정하여 Multi Value Insert로 변환하였습니다.
9
트러블슈팅 — binlog_format 변경
interleaved lock mode(mode=2)는 statement-based replication 사용 시 안전하지 않을 수 있습니다. binlog_format을 MIXED → ROW로 변경하여 replication 안정성을 확보하였습니다. (MySQL 8.0 이후 기본값)

주요 설계 전략

01. 클린 아키텍처 리펙토링

DB 교체 경험을 계기로 도메인의 외부 독립성 확보를 위해 도입하였습니다. 정의된 각 계층들을 멀티모듈로 구성하여 의존성 관리를 강화하였습니다.

Domain — 순수한 도메인 객체들을 가지며, 외부 프레임워크나 라이브러리를 일체 의존하지 않습니다.
UseCase — Domain에 의존하여 애플리케이션에 특화된 규칙들을 정의하고, 도메인들을 가져와서 활용합니다.
Port — Adapter와 UseCase 간의 데이터를 중계하는 인터페이스 역할을 합니다.
Adapter — 웹, 영속성 관련 프레임워크나 라이브러리 등이 위치한 계층입니다. (InAdapter: 요청 수신, OutAdapter: DB 영속화)
  • DIP를 활용해 Adapter와 UseCase 계층 사이의 결합도를 느슨하게 유지
  • Domain/Port 계층은 외부 라이브러리 의존성 없음 — 비즈니스 로직의 안정성과 재사용성 보장
02. 도메인 주도 개발 (DDD)

도메인 응집도를 높여 외부 시스템&요구사항이 변경되더라도 소프트웨어의 품질을 보다 안정적으로 유지할 수 있도록 도메인 주도 설계 방식으로 개발을 진행하였습니다.

순수 도메인 객체 | 영속성 테이블 객체

도메인을 외부로부터 온전하게 보호하기 위해 도메인 규칙 및 로직 관리의 책임을 지닌 순수한 도메인 객체와, 영속성의 책임을 지닌 테이블 객체로 분리하였습니다.

  • 도메인 클래스는 도메인 개념의 명확한 이해를 위해 Value Object를 사용
  • 영속성 테이블 클래스는 실제 테이블에 삽입되는 형식과 일치하도록 모두 Raw Field로 관리
  • 필요에 따라 도메인 클래스에서 VO로 정의된 개념이 테이블 클래스에서는 별도 엔티티로 분리하기도 함
간접 참조 방식

각 테이블 객체 간 의존성을 분리하여 독립적으로 보호하고 외래키로 인한 DB의 오버헤드를 방지하고자 직접참조가 아닌 간접참조 방식을 택했습니다.

도메인 객체 생성 패턴
  • 생성자의 AccessLevel을 Private 타입으로 주어 외부 접근을 차단
  • withId(), withoutId() 팩토리 메서드로 의도에 맞는 완전 객체만 생성되도록 강제
  • 도메인 내부에서 생성 규칙을 안전하게 관리
Event의 활용

도메인 간 결합도를 낮추기 위해, 연계된 처리가 필요할 경우 Event를 활용하였습니다. 또한 이벤트 처리 시, 의도한 요구사항에 따라 트랜잭션 및 에러 핸들링을 처리하였습니다.

Loaded 패턴 — 도메인 변경의 영속화 책임 분리

클린 아키텍처에서 도메인은 영속성 계층을 모르므로, 도메인 객체의 변경을 어떻게 DB에 반영할 것인가가 문제였습니다. UseCase에서 직접 save를 호출하면 영속성 관심사가 비즈니스 로직에 침투하게 됩니다.

이를 해결하기 위해 Loaded<D> 래퍼 클래스를 설계하였습니다.

public class Loaded<D> {
    private final D domain;
    private final Consumer<D> syncCallback;

    public void update(Consumer<D> updater) {
        updater.accept(domain);       // 도메인 변경
        syncCallback.accept(domain);  // 영속성 동기화 (콜백)
    }
}
  • Adapter에서 Loaded 생성 시, syncCallback에 JPA 엔티티 동기화 로직을 주입
  • UseCase에서는 loadedMate.update(mate -> mate.close()) 한 줄로 도메인 변경 + 영속화가 완료
  • UseCase가 영속성 메서드(save)를 직접 호출하지 않으므로 계층 간 책임이 명확하게 분리됨
  • syncCallback으로 도메인 → JPA 엔티티 필드 동기화 후, JPA 더티체킹이 트랜잭션 커밋 시 자동으로 DB에 반영

트러블 슈팅

메이트 신청 정원 초과 동시성 이슈 해결
Problem

선착순 모집 시 동시 신청으로 정원을 초과하여 승인되는 Race Condition 발생

Solution

낙관적 락으로 충돌 감지 + 자동 재시도 (최대 3회, 100ms backoff)

동작 흐름
유저A: load(v=1) → 승인 → commit(v=2) ✅ 성공
유저B: load(v=1) → 승인 → commit 시 version 불일치!
    → 자동 재시도 → load(v=2) → 정원 초과 → 거절
의사 결정 과정
1. 비관적 락 (SELECT FOR UPDATE)

동시성을 확실하게 보장하지만, 같은 행에 동시 요청이 들어오면 락이 해제될 때까지 대기가 발생합니다. 메이트 신청은 충돌률이 상대적으로 낮은 기능이라 판단하여 배제하였습니다.

2. 원자적 업데이트 (Atomic Update)

락 점유 시간이 최소라는 장점이 있지만, 메이트 신청 로직은 카운트 증감 외에도 신청 레코드 생성, 알림 발행 등이 하나의 트랜잭션에서 이루어지므로, 카운트만 보호하는 원자적 업데이트로는 전체 로직의 정합성을 보장할 수 없었습니다.

3. 낙관적 락 ✅ (선택)

데드락 위험이 없고, 트랜잭션 전체(정원 체크 + 신청 레코드 생성 + 카운트 증감 + 알림 발행)를 보호할 수 있습니다. 메이트 신청은 같은 모집에 동시 신청하는 빈도가 낮은 저경합 환경이므로, 충돌이 드물어 재시도 비용도 미미하다고 판단하여 선택했습니다.

재시도 전략: 최대 3회, 100ms backoff

충돌 시 서버에서 자동 재시도합니다. 재시도 간 100ms 지연을 두어 상대 트랜잭션의 커밋 시간을 확보했고, 횟수를 너무 높이면 응답 지연이 길어지므로 저경합 환경에서 충분한 최대 3회로 제한하였습니다. 동시성 테스트를 작성하여 정원 초과가 발생하지 않음을 검증하였습니다.
